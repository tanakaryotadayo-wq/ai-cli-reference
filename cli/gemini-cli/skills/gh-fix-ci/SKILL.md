# GH Fix CI (CI自動修復ルール)

GitHub Pull Request に紐付く GitHub Actions などの CI 実行結果から失敗ログを取得・解析し、原因特定と修復プランの提案・適用を行う。

## 前提条件

`gh auth login` で認証済みであること（`repo` および `workflow` スコープが必要）。

## Golden Rule (曖昧性収縮定理)

CI 失敗の原因ログのパースに失敗した場合、または原因や修復方針が複数考えられて曖昧さがある場合は、独自の当てずっぽうの修正をコードに適用してはならない (MUST NOT)。必ず具体的なエラーログのスニペットとともに、複数の修復プランをユーザーに提示し、明示的な承認を得なければならない (MUST)。

## Stop Rule (散逸停止定理)

修正適用後のテスト実行エラー、`gh pr checks` による再検証エラー、または Git プッシュコマンドのエラーが連続して **5回以上** 発生した場合は、自動修復の無限ループを防ぐため、直ちに処理を強制停止し、エラーログを要約してユーザーの指示を仰がなければならない (MUST)。

## Task Execution Workflow (最小作用ワークフロー定理)

CI 修復を行う際、エージェントは以下の手順を厳格に実行しなければならない (MUST)。

1. **認証とPRの特定**: `gh auth status` で認証を確認し、カレントブランチの PR 番号または URL を特定する。
2. **失敗チェックの検査**: `gh pr checks` コマンドで失敗しているジョブを特定し、該当する Actions の run ID のログ、または進行中ジョブのログを取得する。
3. **失敗原因の要約**: 失敗したチェック名、エラー箇所の前後の簡潔なログスニペット（マスク処理済）、および run URL を整理してユーザーに報告する。
4. **修復プランの提示と承認**: 修正すべき箇所と具体的な修正コードを含む修復プランを作成し、実行の承認を要求する。
5. **修正の実装と再検証**: 承認が得られた場合に限りコードを変更し、ローカルでテストが正常に通ることを確認した上でプッシュし、再び `gh pr checks` で CI がパスしたかを再検証する。
