# GH Address Comments (GitHub コメント対応ルール)

GitHub Pull Request のレビューコメントや issue コメントを取得し、ユーザーが選択した特定のコメントに対してのみ修正を適用する。

## 前提条件

`gh auth login` で認証済みであること。`gh auth status` で確認。

## Golden Rule (曖昧性収縮定理)

コメントの指摘内容、または修正すべきファイルやコードの具体的な修正方法に複数の解釈があり曖昧な場合は、独自の憶測でコードを書き換えてはならない (MUST NOT)。必ずユーザーに質問するか、修正ドラフトを提示して明示的な確認を得なければならない (MUST)。

## Stop Rule (散逸停止定理)

`gh` コマンドの実行エラー（ネットワークエラー、認証切れ等）、またはファイルの編集・Gitコミット・プッシュ時におけるエラーが連続して **5回以上** 発生した場合は、即座に処理を中断し、エラーの原因となったコマンドとエラーログを要約して異常終了しなければならない (MUST)。

## Task Execution Workflow (最小作用ワークフロー定理)

コメント対応を実行する際、エージェントは以下の手順を厳格に実行しなければならない (MUST)。

1. **前提とコメントの取得**: `gh auth status` で認証を確認し、対象PRの全コメントおよびレビュースレッドを取得する。
2. **番号付き一覧の提示**: 取得したコメントとレビュースレッドに一意の番号を振り、それぞれの修正概要とともにユーザーに一覧を提示する。
3. **ユーザーによる選択の待機**: ユーザーに「どの番号の指摘に対応するか」を問い、選択を待つ。
4. **選択されたコメントのみの修正**: ユーザーが選択した番号のコメントに対応するコードのみを修正し、テストを実行した上でコミットおよびプッシュを行う。選ばれなかったコメントには手を触れてはならない (MUST NOT)。
5. **認証・接続エラーのリカバリ**: `gh` コマンドで認証エラーやレート制限を検知した場合は、再認証手順をユーザーに提示して指示を仰ぐ。
