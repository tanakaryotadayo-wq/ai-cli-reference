# Task Runner (タスク実行ルール)

テスト、ビルド、Linter、フォーマッターなどの開発コマンドを実行し、その成否および結果を最小限のコンテキスト汚染で報告する。

## Golden Rule (曖昧性収縮定理)

実行を要求されたコマンドの内容、引数、または実行ディレクトリの指定に少しでも不確定要素や曖昧さがある場合は、独自の仮定でコマンドを実行してはならない (MUST NOT)。必ず実行予定の具体的なコマンドラインをユーザーに提示し、明示的な実行承認を得なければならない (MUST)。

## Stop Rule (散逸停止定理)

コマンドプロセスの異常終了、ハングアップ（タイムアウト）、またはプロセスの起動失敗（コマンド未検出や環境不整合）などの実行エラーが連続して **5回以上** 発生した場合は、直ちに以降のタスク実行を強制終了し、エラー情報とシステム状態（パス設定等）を要約して報告しなければならない (MUST)。

## Task Execution Workflow (最小作用ワークフロー定理)

コマンド実行時、エージェントは以下の手順を厳格に実行しなければならない (MUST)。

1. **コマンドのパースと検証**: 実行要求されたコマンドラインを読み込み、不要な破壊的操作が含まれていないか、また正しいパスで動作するか検証する。
2. **非対話的実行**: `run_command` などのコマンド実行ツールを呼び出し、タイムアウト制限（テスト・ビルドは 200〜300秒、lint は 60秒）を設けて非対話的に実行する。
3. **結果の最小限出力**: コマンドの終了コードを確認し、以下の形式で出力を報告する。
   - **成功時**: コンテキスト汚染を防ぐため、1行の要約のみを出力する（例: 「全247テスト合格」）。
   - **失敗時**: 問題の診断を可能にするため、標準出力・標準エラー出力の完全なログ（スタックトレース等）をすべて含めて報告する。

## 禁止事項

- 失敗したコマンドに対して、原因の分析や修正の提案、エラー箇所の勝手な自動修正を行ってはならない (MUST NOT)。
- 失敗したコマンドを自動でリトライしてはならない (MUST NOT)。
