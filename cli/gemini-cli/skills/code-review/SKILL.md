# Code Review (コードレビュー)

コード変更を精査し、重大な不具合（バグ、脆弱性、競合状態等）のみをピンポイントで報告する。

## 指導原則

フィードバックは「洗濯後のジーンズから2000円札を見つけた」級の本物の価値だけ。ノイズは出すな。スタイルや命名規則、ドキュメントの欠如などの些細な問題は完全に無視せよ。

## Golden Rule (曖昧性収縮定理)

レビュー中に検出された問題の確信度が低い場合、または不具合かどうか判断に迷う仕様の曖昧さがある場合は、独自の憶測で警告を乱発してはならない (MUST NOT)。「確信が持てないなら言及しない」を原則とし、安全側に倒して「問題なし」と判定しなければならない (MUST)。

## Stop Rule (散逸停止定理)

Git コマンドの実行エラー、またはファイルの読み込み・構文解析等の処理エラーが連続して **5回以上** 発生した場合は、システムの暴走を防ぐため即座にレビュープロセスを強制停止し、そこまでに発生したエラーサマリーを報告しなければならない (MUST)。

## Task Execution Workflow (最小作用ワークフロー定理)

レビュー実行時、エージェントは以下の手順を厳格に実行しなければならない (MUST)。

1. **変更範囲の把握**: `git status` や staged/unstaged diff 等を確認し、レビューすべき具体的なコードの変更範囲を特定する。
2. **コンテキストの理解**: 変更箇所の周辺コードを読み込み、設計目的、システムとの統合関係、および不変条件を理解する。
3. **欠陥の精査**: 以下の重大な観点に絞ってコードをレビューする。
   - 致命的なバグ・ロジックエラー
   - セキュリティ脆弱性・インジェクションリスク
   - 競合状態や並行性の問題
   - メモリリークやリソース管理の欠陥
   - 未処理の例外・クラッシュ要因
4. **高確信の問題のみ報告**: 確実な問題のみを出力フォーマットに従って出力する。重大な問題がない場合は、「レビューした変更に重大な問題は見つかりませんでした。」のみを出力して終了する（コードを褒めたり要約したりしてはならない）。
