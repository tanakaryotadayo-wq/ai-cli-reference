## 2. 検出された欠陥の詳細リスト

### 指摘 1: 一時ファイルのクリーンアップ手順における MUST 表記の漏れ (レイヤー: `L1`, 深刻度: `高`)
- **位置**: `1. 動作要件 - データ出力 (46行目)`
- **分類**: `rfc2119_missing_must`
- **ダメだし (criticism)**: アトミックな書き込み処理において、エラー発生時の一時ファイル削除（クリーンアップ）はファイルシステムの不要なゴミ蓄積や予期せぬ名前衝突を防ぐために必須の要件です。しかし、46行目の「4) 処理中にエラーが発生し中断した場合は、作成中の一時ファイルを確実に削除（クリーンアップ）すること。」には (MUST) 表記が欠落しています。これにより、クリーンアップ処理の実装が任意と解釈され、残存した一時ファイルが次回実行時のアトミック書き込みや競合を阻害するリスクが生じます。
- **改善点 (improvement)**: 一貫性を保ち、エラーハンドリングの実装漏れを防ぐために、46行目の末尾に明確に (MUST) を追加する必要があります。
- **修正案**:
  ```diff
  4) 処理中にエラーが発生し中断した場合は、作成中の一時ファイルを確実に削除（クリーンアップ）すること (MUST)。
  ```

- **修正案**:

### 指摘 4: ロックファイル取得時のノンブロッキング化と自動リセットの安全設計 (レイヤー: `L2`/`L6`/`L9`, 深刻度: `高`)
- **位置**: `## 3. 動作安全性 - 再帰呼び出しガード (113-119行目)`
- **分類**: `deadlock_risk`
- **ダメだし (criticism)**: 共有一時ファイルに対する `fcntl.flock` での排他ロックの取得方法が指定されていない。もしブロッキングで取得した場合、並行実行された別プロセスが永久に待機状態になりハングアップ（デッドロック）する。また、ロックを獲得する前に生存確認（プロセス存在確認や5分経過の判定）を行うことは、ロックされていない状態でのファイル読み書きによる競合を招き、逆にロック獲得後に確認を行うと、デッドロックしている場合に生存確認コード自体が実行されないため自動リセットが機能しなくなる。
- **改善点 (improvement)**: ロック取得時はノンブロッキング（`LOCK_EX | LOCK_NB`）で試行し、ロックが取得できなかった場合に初めて生存確認（PID存在チェックやタイムスタンプの経過判定）を行い、安全にリセットし直す状態遷移を規定する。また、ロックファイルへの書き込みはオープン直後の獲得クリティカルセクション内でアトミックに行い、書き込み後は `flush` と `os.fsync` で物理的に書き込みを保証する。
- **修正案**:
  ```markdown
  スクリプトの起動直後に、共有一時ディレクトリ内の実行カウンタファイル `$APP_DATA_DIR/scratch/execution_depth.lock` を読み書きモード（新規作成を許可）でオープンし、直ちに排他ロック（`fcntl.flock`による `LOCK_EX | LOCK_NB`）を取得すること (MUST)。もしロックの取得に失敗した場合は、ロックファイルから PID とタイムスタンプを読み取り、該当プロセスが存在しないか、または作成から 5 分以上経過している場合に限り、ロックファイルを強制的に削除・再作成してロックを再取得する生存確認・自動リセットロジックを実装すること (MUST)。
  ```

### 指摘 6: ルート外シンボリックリンク読み取りによる情報漏洩（LFI）の脆弱性 (レイヤー: `L4`, 深刻度: `高`)
- **位置**: `## 3. 動作安全性 (Safety Rules) / パスの動的解決とバリデーション (124行目)`
- **分類**: `arbitrary_file_read_via_symlink`
- **ダメだし (criticism)**: 「読み取り専用の監査対象ファイル（ソースコード等）のパスについては、プロジェクトルート外であってもシンボリックリンクを含めて正規化・解決し、読み取りを許可すること」と規定されています。しかし、この設計は悪意のあるシンボリックリンクを介して、システム上の任意のファイル（SSH秘密鍵、パスワードファイルなど）を監査スクリプトが読み込み、漏洩するリスク（LFI脆弱性）を発生させます。
- **改善点 (improvement)**: プロジェクトルート外のファイルの読み取りを許可する場合でも、シンボリックリンクのリンク先（ターゲット）が安全な許可ディレクトリ（カレントワーキングディレクトリ、プロジェクトのルート）の配下に収まっているかを Path.resolve() で検証し、その範囲外のシンボリックリンク解決はエラーとして却下するバリデーションルールを導入すべきです。
- **修正案**:
  ```diff
  + 読み取り専用の監査対象ファイル（ソースコード等）のパスについては、プロジェクトルート外の参照を許可する場合でも、シンボリックリンクの解決先（リンク先の実体パス）が、事前に定義された「安全な監査対象スコープ（プロジェクトルート等）」の配下にあることを厳密に検証することとし、スコープ外の機密ファイル（システムファイルや認証鍵等）へのシンボリックリンク解決はセキュリティ例外として却下すること (MUST)。
  ```

### 指摘 9: 反例ファイルリスト制限時の `counterexamples_truncated` と先着順ポリシーの追加 (レイヤー: `L6`, 深刻度: `高`)
