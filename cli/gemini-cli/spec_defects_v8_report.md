## 2. 検出された主要な欠陥（致命的・高）の要約

### 致命的指摘 1: 再帰呼び出しガード（排他ロック）の早期解放による並行競合 (レイヤー: `L2`/`L4`/`L6`/`L9`)
- **位置**: `3. 動作安全性 -> 再帰呼び出しガード (138-143行目)`
- **ダメだし (criticism)**: スクリプト起動時にロックファイルを取得し、カウンタ値をインクリメントした直後にロックを解放して本処理（統計検証等）に入っていました。このため、本処理実行中に別プロセスが起動した場合に排他制御が効かず、カウンタの競合破壊や無限ループ防止機能の無効化を招きます。また、初回起動時の空ファイルに対する整数変換（パースエラー）の考慮漏れや、終了時のデクリメント・削除処理で `flock` を用いた排他制御を行わずにファイルを上書き・削除するため、並行プロセス終了時の状態破壊リスクがありました。
- **改善点 (improvement)**: 排他ロックは、スクリプト実行開始から `finally` ブロックでのカウンタデクリメント（および削除）が完了するまでの全実行期間中保持し続ける仕様に変更します。また、初回空データ時はカウンタ `0` を適用し、デクリメントおよび削除操作時もアトミックなロック保護を適用します。

### 致命的指摘 2: JSONパースエラー時の縮退運転継続と一律異常終了の論理矛盾 (レイヤー: `L2`/`L9`)
- **位置**: `3. 動作安全性 -> Golden Rule (デフォルト動作と縮退運転) (129-133行目)`
- **ダメだし (criticism)**: 仕様書では「パースエラー率が5%以下ならスキップして続行」としつつ、同時に「JSONパース自体に失敗した場合は一律で全定理検証不可として終了コード2で異常終了する」と規定していました。これにより、1件でもデータ破損が発生しただけでしきい値に関わらず異常終了してしまい、「縮退運転（Graceful Degradation）」が完全に崩壊していました。
- **改善点 (improvement)**: 5%以下の軽微な JSON パースエラーは単に警告ログを出してスキップし、残りの正常データで検証処理（縮退運転）を継続させ、5%を超えた場合に初めてすべての定理を `UNVERIFIABLE`（検証不可）として異常終了するよう論理境界を整理します。

### 致命的指摘 3: エージェント実行・リカバリ操作における Stop Rule の定義欠落 (レイヤー: `L5`)
- **位置**: `3. 動作安全性 -> Stop Rule (134-137行目)`
- **ダメだし (criticism)**: スクリプト内部の低レイヤーな I/O 例外リトライしか定義されておらず、エージェントが実行するリカバリ操作（mvやrm等のコマンド実行、ファイル編集の競合）における停止基準が定義されていませんでした。エラーが連続発生した場合に無限ループに陥り、ファイルシステムの破損やリソースの浪費を招くリスクがありました。
- **改善点 (improvement)**: コマンド実行やファイル編集の競合エラーが連続して 5 回以上発生した場合は、即座にすべての自動処理・変更操作を停止し、エラーログを要約してオペレータの指示を仰ぐという「Stop Rule」を追加します。

### 致命的指摘 4: 自己反映ループにおけるプロンプトインジェクション脆弱性 (レイヤー: `L9`/`L4`)
- **位置**: `1. 動作要件 -> JSONメタデータ出力 (63-66行目)`
- **ダメだし (criticism)**: `defects.jsonl` 内のテキストデータ（`criticism` や `improvement` 等）に悪意あるプロンプト（指示無視プロンプト等）が含まれていた場合、出力された JSON を読み込む後続の AI エージェントがこれを「システム命令」として誤認して実行する「セカンドオーダー・プロンプトインジェクション」のリスクが極めて高い状態でした。
- **改善点 (improvement)**: 動的テキストデータを XML タグ（例: `<untrusted_metadata>`）などで厳格にカプセル化し、システムプロンプトのコンテキストとデータを完全に分離して解釈させる安全な設計要件を定義します。

### 高指摘 1: 欠陥数ゼロ時の再現性判定における論理的パラドックス (レイヤー: `L2`/`L1`)
- **位置**: `1. 動作要件 -> 定理の統計的検証 (22-25行目)`
- **ダメだし (criticism)**: 再現性の条件として「30%以上のファイルで検出されること」を要求していますが、「欠陥数ゼロ（健全）」の例外合格時にもこの数式が適用されると、0%となり不合格になってしまう論理的パラドックスがありました。
- **改善点 (improvement)**: 当該定理に対応する欠陥が全ファイルで1件も検出されなかった場合（分母が0の場合）は、再現性判定を自動合格とみなす例外条件を追加します。

### 高指摘 2: 反例履歴バッファ上限による情報の切り捨て (レイヤー: `L6`)
- **位置**: `1. 動作要件 -> メモリ制限 (18-19行目)`
- **ダメだし (criticism)**: バッファ上限（1,000件）を超えたら先着順で切り捨てるだけの設計は、特定のファイルがメモリを独占して以降の反例が完全に無視・不可視化される問題がありました。
- **改善点 (improvement)**: 1,000件を超える反例パスについては、`0600` 権限の一時ファイルにストリーム出力して退避（コンパクション）させ、最終レポート時にマージする仕様へ変更します。

### 高指摘 3: 異常終了時における HITL メタデータ伝達プロトコルの欠落 (レイヤー: `L7`)
- **位置**: `2. 動作安全性 -> 異常終了時ログ`
- **ダメだし (criticism)**: スクリプトが異常終了した際、上位エージェントが「何を自動修復し、何をオペレータに承認（HITL）させるべきか」をログから機械的に判別する構造化シグナルが存在しませんでした。
- **改善点 (improvement)**: 異常終了時のエラーログ内に、具体的なリカバリ指示を示すメタデータとして `hitl_recovery_context`（`recovery_action` [例: `RESTORE_BACKUP`, `REMOVE_LOCK_FILE`]、および `target_path`）を必ず含めるよう仕様を追加します。
