# verify_theorems.py 仕様書 (v7)

## 概要
AIエージェント工学の3つの定理候補（散逸停止定理、曖昧性収縮定理、最小作用ワークフロー定理）について、すでに監査・検出された欠陥データ（`defects.jsonl`）および対象ファイルリストを用いて、定理の成立条件（再現性、影響度、反例比率）を統計的に集計・検証する Python スクリプトの実装仕様書。

各定理の最終的な「合格（成立）」判定は、再現性、影響度、反例比率のすべての検証基準を同時に満たした場合（論理積：AND）のみ合格とし、いずれか一つでも満たさない場合は不合格とすること (MUST)。

また、検証結果のレポート出力および監査エージェントの指摘スキーマにおいて、忖度なしの「ダメだし (criticism)」と「改善点 (improvement)」を出力することを義務付ける (MUST)。

## 1. 動作要件 (Functional Requirements)

### データ入力
- **データ入力**: コマンドライン引数 `--defects` で指定されたパス（デフォルト: `$APP_DATA_DIR/scratch/defects.jsonl`）から欠陥データファイル（`defects.jsonl`）を1行ずつ読み込み、パースすること (MUST)。
- **入力サイズ制限（DoS対策）**: `defects.jsonl` を1行ずつ読み込む際、1行あたりの最大許容サイズ（10MB）を設け、これを超える巨大な行を検出した場合は、セキュリティ保護のため処理を中断し、終了コード `1` で異常終了すること (MUST)。
- **メモリフットプリント制限**: パース後の詳細なオブジェクトをすべてメモリ上に保持してはならず (MUST NOT)、集計用の最小限のメトリクス（正規化されたファイルパスごとの `pattern_class` カウント、深刻度比率、レイヤーごとの深刻欠陥の有無、および反例判定フラグ・反例ファイルリスト）のみを状態変数としてメモリに保持すること (MUST)。
  - ただし、メモリに保持する反例ファイルパスの最大件数（履歴バッファ上限）は 1,000件 とし (MUST)、これを超える件数の反例が検出された場合は、リストへの追加を停止して件数のみをカウントすること (MUST)。これにより大規模データ処理時のメモリ枯渇を防止する。

### 定理の統計的検証
- **再現性 (Reproducibility)**: 各定理に対応する以下の `pattern_class` が、監査対象の全ファイル（読み込まれたパスリストの総件数。デフォルト88件）中 **30%以上** で検出されていることを検証すること (MUST)。
  - 散逸停止定理: `missing_stop_rule` であることを検証すること (MUST)。
  - 曖昧性収縮定理: `missing_golden_rule` であることを検証すること (MUST)。
  - 最小作用ワークフロー定理: `missing_execution_workflow` または `missing_workflow` のいずれか一方（あるいは両方）が1件以上検出されているファイルを対象とし、該当するユニークなファイル数の論理和（OR）として分子をカウントし、検出率を計算すること (MUST)。
- **影響度 (Impact)**: 各定理に対応する検出された欠陥のうち、深刻度（`severity` フィールドの値が `"高"` または `"致命的"`、あるいは `"high"` または `"critical"`）である割合（深刻度比率）が **50%以上** であるかを検証すること (MUST)。
  - 最小作用ワークフロー定理においては、`missing_execution_workflow` と `missing_workflow` の欠陥を同一の定理カテゴリとして合算（マージ）した欠陥集合を対象として集計を行うこと (MUST)。
  - ただし、対応する欠陥が1件も検出されていない場合は、ゼロ除算を防止するため深刻度比率を 0% として処理し、当該定理は基準を満たしていない（不合格）と判定すること (MUST)。
- **反例検証 (Counterexample Analysis)**: 各定理に対応する `pattern_class` が検出されている（＝ルールが未定義である）ファイル群を分母とし、その中で、L2 または L4 レイヤー（`layer` フィールドの値が `"L2"` または `"L4"`）かつ深刻度（`severity` フィールドの値が `"高"` または `"致命的"`）である欠陥が1件も検出されていないファイル（＝反例）の割合（分子 / 分母）が **10%以下** であるかを判定すること (MUST)。
  - **欠陥数ゼロの例外処理**: ただし、分母が0（該当する欠陥が1件も検出されていない）の場合は、プロジェクトが当該カテゴリについて完全に健全であるとみなし、再現性・影響度・反例比率のすべてについて暫定的に基準を満たしたもの（合格）として判定すること (MUST)。この場合、健全な合格であるため、自動修復のための HITL 警告（`hitl_required` = `false`）とすること (MUST)。

### ダメだし (criticism) と 改善点 (improvement) の義務化
- **検証プログラムの出力結果における要件**:
  - `theorem_verification_results.json` (JSON) および `theorem_verification_results.md` (Markdownレポート) において、各定理の判定結果を出力すること (MUST)。
  - JSONファイル内では、キー名として `criticism` および `improvement` を使用すること (MUST)。外部のLLM呼び出し等は行わず、スクリプト内の数値条件分岐に基づいたルールベースの定型テキスト生成ロジックとして完結させること (MUST)。
  - Markdownレポート内では、見出し名として「ダメだし (Criticism)」および「改善点 (Improvement)」を使用すること (MUST)。
    - **`criticism` (ダメだし)**: 対象定理の検証基準が未達成となった理由や、データに含まれる具体的な欠陥傾向（なぜ反例が多いのか、なぜ深刻度比率が低いのかなど）について、集計されたメトリクス（検出率、深刻度比率、反例比率）の値に応じて事前に定義されたテンプレート文から動的に生成する客観的な指摘文章。
    - **`improvement` (改善点)**: 検証基準をクリアするために、AIエージェントの行動仕様、設計ルール、またはデータ収集プロセスをどのように変更すべきかを、各メトリクスの未達成状況に応じた具体的なアクションアイテムとして提示する文章。
  - 定理が合格（成立）している場合でも、さらなる堅牢性向上の観点から「ダメだし (criticism)」と「改善点 (improvement)」を記載すること (MUST)。

### データ出力
- **レポート出力の整合性保護**: 検証結果のメトリクス、合格/不合格判定、および「ダメだし」「改善点」をまとめたレポートを、コマンドライン引数 `--output` で指定された出力パス（デフォルト: `$APP_DATA_DIR/scratch/theorem_verification_results.md`）に書き出すこと (MUST)。
- **ファイル出力の状態遷移シーケンス**: ファイル出力（MarkdownおよびJSON）を行う際は、ターゲットパスおよびバックアップパスがシンボリックリンクでないこと（`is_symlink()` が偽であること）(MUST)、および解決された絶対パスが許可されたディレクトリ（`$APP_DATA_DIR` の配下、またはプロジェクトルートや作業ディレクトリ配下）にあることを検証した上で、以下の順序でアトミックに実行すること (MUST)。
  1) 一時ファイル（ターゲットパスに拡張子 `.tmp` を付与したもの）に検証結果を書き出すこと (MUST)。
  2) 書き出しが成功した場合、既存のレポートファイルが存在すれば、それがシンボリックリンクでないことを確認した上で、バックアップパス（ターゲットパスに拡張子 `.bak` を付与したもの）へ「コピー（コピー完了まで元のファイルを削除しない）」して退避させること (MUST)。
  3) 一時ファイルをターゲットパスへアトミックにリネーム（置換）すること (MUST)。
  4) 処理中にエラーが発生し中断した場合は、作成中の一時ファイルを確実に削除（クリーンアップ）すること (MUST)。
- **ファイルパーミッション制限**: レポートファイルおよび一時ファイルを作成する際は、所有者のみ読み書き可能なアクセス権限（`0600`）を明示的に指定して作成すること (MUST)。出力先ディレクトリが存在しない場合は、所有者のみ読み書き実行可能な権限（`0700`）で自動作成すること (MUST)。
- **インジェクション防止**: レポート生成時、`defects.jsonl` から取得したテキストデータを埋め込む際は、マークダウンの特殊文字（`\`, `\``, `*`, `_`, `{`, `}`, `[`, `]`, `(`, `)`, `#`, `+`, `-`, `.`, `!`）の直前にバックスラッシュ `\` を挿入してエスケープし (MUST), かつ HTML 特殊文字（`&`, `<`, `>`, `"`, `'`）を Python の `html.escape` を用いて実体参照に置換して無効化すること (MUST)。
- **HITL警告**: 検証において、無効データの多発やデータの破損等、例外的な処理エラーにより「分母が0」となったことが疑われる場合に限り、Markdownレポート内にオペレータの確認が必要な旨の警告項目「HITL確認要求」を明記し (MUST)、`structlog` を用いて `hitl_required=True` メタデータを含む警告ログ（`WARNING` レベル）を標準出力に出力し、オペレータに確認を促すこと (MUST)。単に欠陥検出数が 0 件である健全な合格ケースでは、警告を出力してはならない (MUST NOT)。
- **JSONメタデータ出力**: 後続のAIエージェントや検証パイプラインが自己反映（Self-Reflection）および自動修復のアクションを実行できるよう、検証結果の詳細（各定理の判定、メトリクス数値、反例ファイルリスト、`criticism`、`improvement`、および `hitl_required` (Boolean型)）を含む構造化された JSON ファイルを、コマンドライン引数 `--output-json` で指定されたパス（デフォルト: `$APP_DATA_DIR/scratch/theorem_verification_results.json`）に書き出すこと (MUST)。
  - `hitl_required` フラグが `true` の場合、後続の自動修復パイプラインは自動実行を中断し、オペレータの確認を待つこと (MUST)。
  - `hitl_required` が `true` の場合は、`structlog` にて `event='HITL_PENDING_APPROVAL'` という一意のイベント名を持つ構造化ログを出力し、後続のパイプラインやCI/CD環境がそのログイベントをキャッチして Slack や GitHub PR 等に承認依頼を自動通知できるようにシグナリング仕様を統合すること (MUST)。
- **非対話的実行原則**: スクリプトの実行は常に非対話的（Non-blocking）に行うものとし、警告ログを出力した後にオペレータの入力を待って一時停止してはならない (MUST NOT)。警告は出力およびファイル記録のみにとどめること (MUST)。

## 2. Task Execution Workflow (実行手順)

### オペレーター/エージェント作業ワークフロー

#### 1. 事前準備 (MUST)
1. 監査対象のパスリストが記述された JSON ファイル `target_files.json` を作成し、プロジェクトのルートディレクトリまたは指定された作業ディレクトリに配置すること (MUST)。
2. 欠陥データ `$APP_DATA_DIR/scratch/defects.jsonl` が存在し、読み取り可能であることを確認すること (MUST)。
3. 実行環境に `uv` がインストールされており、仮想環境が有効化されていることを確認すること (MUST)。

#### 2. 実行コマンドと引数指定 (MUST)
1. スクリプトは以下のコマンドで非対話的に実行すること (MUST)。
   ```bash
   uv run verify_theorems.py \
     --targets target_files.json \
     --defects $APP_DATA_DIR/scratch/defects.jsonl \
     --output $APP_DATA_DIR/scratch/theorem_verification_results.md \
     --output-json $APP_DATA_DIR/scratch/theorem_verification_results.json
   ```
2. デバッグ時は `--debug` オプションを付与し、処理の各ステップやパース詳細をログ出力すること (MUST)。

#### 3. 実行中進捗の監視 (MUST)
1. 標準出力に `structlog` から出力される JSON 形式のログを監視すること (MUST)。
2. 進捗フェーズ（`\"setup\"`, `\"mapping\"`, `\"statistics\"`, `\"reporting\"`）の遷移が正しく行われ、最終的に `\"reporting\"` フェーズが完了することを確認すること (MUST)。

#### 4. 異常終了時のリカバリ手順 (MUST)
スクリプトが異常終了した場合、オペレータは終了コードに応じて以下の手順を実行すること (MUST)。
- **終了コード 1（I/Oエラー、セキュリティ違反、無効引数）**:
  1) `structlog` のエラーログからエラー原因のファイルパスおよびセキュリティバリデーションエラーの内容を確認する。
  2) 対象パスが `$APP_DATA_DIR` 等の許可ディレクトリ内にあるか確認する。
  3) ディレクトリの書き込み権限や空き容量を修正する。
  4) アトミック書き込みの過程でリネーム（置換）に失敗し、既存のレポートファイルが破損または消失した疑いがある場合は、バックアップファイル（`.bak`）から手動で復元（例: `mv report.bak report`）を実行する。
- **終了コード 2（パースエラーしきい値超過）**:
  1) ログおよび最終レポート（出力されている場合）の「無効データ」セクションから、エラーとなった行番号と生データを確認する。
  2) `defects.jsonl` の該当箇所を修正して JSONL のフォーマットを満たすようにする。
- **終了コード 3（再帰制限超過）**:
  1) スクリプトが無限ループや無制限の再起動に陥っていないかログを確認する。
  2) 安全を確認した上で、一時ディレクトリのロックファイル `$APP_DATA_DIR/scratch/execution_depth.lock` を手動で削除し、実行カウンタをリセットする。

## 3. 動作安全性 (Safety Rules)

- **構造化ログの適用**: スクリプト内のすべてのログ出力（進捗、デバッグ、エラー）は、`structlog` を使用して構造化 JSON 形式で出力すること (MUST)。標準の `logging` や `print` は使用してはならない (MUST NOT)。
- **情報のフィルタリング（マスク処理）**: ログ出力処理において、環境変数や引数から取得したパス、シークレットなどの機密情報がそのままログに出力されないよう、起動時に動的にカレントユーザー名（例: `getpass.getuser()` で取得される文字列）や `APP_DATA_DIR` のパスを検出し、`structlog` のカスタムプロセッサを通じて `[USER]` に置換するマスク処理や、機密情報（APIトークン、パスワードなど）を `[REDACTED]` にマスクする処理を実装すること (MUST)。
- **進捗ログの標準化**: 進捗ログはタイムスタンプの出力形式として ISO 8601（UTC または ローカルタイムゾーン付きミリ秒精度）を強制し、`structlog.processors.TimeStamper(fmt="iso")` の使用を指定すること (MUST)。また、進捗フェーズのログでは、`event` フィールドに `\"workflow_phase_changed\"` を設定し、`phase` キーの値（以下のフェーズ定義）を出力すること (MUST)。
  - `\"setup\"`: スクリプト起動、環境変数・引数パース、再帰呼び出しカウンタの検証および初期化処理。
  - `\"mapping\"`: ターゲットファイルリストおよび欠陥データの読み込み、絶対パスへの正規化、およびファイルごとの欠陥マッピング処理。
  - `\"statistics\"`: 各定理に対応する統計メトリクスの集計および検証判定の計算処理。
  - `\"reporting\"`: MarkdownおよびJSONレポートの一時ファイル書き出し、バックアップ作成、アトミック置換、およびカウンタファイルのクリーンアップ処理。
- **Golden Rule (デフォルト動作と縮退運転)**: 
  - データパース時において、JSONデコードエラー、あるいは必須キー（`target`, `pattern_class`, `severity`, `layer`）が欠落している行を「無効データ」と判定し、その行を警告ログ（レベル: `WARNING`）として記録した上でスキップし、残りのデータのパース処理を継続すること (MUST)。警告ログにはエラーが発生した「ファイルパス」「行番号」「生データの抜粋（マスク処理後）」「パース失敗理由（例外メッセージ）」を必ず含めること (MUST)。
  - パース処理の開始前に対象ファイルの総行数（総レコード件数）をプリスキャンして取得し、パース中に検出された無効データ件数の累積がその 5% を超えた時点で、データの健全性が損なわれていると判断し、`structlog` で標準エラー出力に警告ログを出力すること (MUST)。
  - この際、即座に全体を異常終了させるのではなく、影響を受ける特定の定理の検証のみを「データ破損により検証不可」として除外し、正常なデータが存在する他の定理の検証・集計およびレポート出力を継続する「縮退運転（Graceful Degradation）」を行うこと (MUST)。ただし、すべての定理が検証不可となった場合は、それまでに検出された全パースエラーのサマリー（行番号と理由のリスト）を標準エラー出力に一括出力した上で、終了コード `2` で異常終了すること。
- **Stop Rule (異常停止条件と指数バックオフ)**: 
  - ファイルの読み込みまたは書き込みにおいて、回復不能な I/O エラー（`PermissionError`, `FileNotFoundError`）が発生した場合、または一時的な I/O エラー（`BlockingIOError`, `TimeoutError`）が発生した場合は、ジッターを伴う指数バックオフ（初期待ち時間 0.5 秒、乗数 2、最大待ち時間 5 秒、最大リトライ回数 5 回）を適用したリトライループを実装すること (MUST)。
  - リトライ回数を超過した、あるいは 429 レート制限などで即時にフォールバックが必要な場合は、実行を即座に停止し、`structlog` で例外情報（`exc_info=True` を指定し、適切にフォーマットされたスタックトレース）と、マスク処理済みの詳細な診断コンテキスト情報（エラーが発生したファイルパス、実行中のアトミックシーケンスフェーズ、処理中行番号、リトライ試行回数等）を `ERROR` または `CRITICAL` レベルで出力した上で、終了コード `1` で異常終了すること (MUST)。スタックトレース出力時も情報のフィルタリング要件を適用し、シークレットや認証情報を確実にマスクすること (MUST)。
- **再帰呼び出しガード**: 
  - エージェント実行環境下におけるスクリプトの無限再帰呼び出しを防止するため、スクリプトの起動直後に、共有一時ディレクトリ内の実行カウンタファイル `$APP_DATA_DIR/scratch/execution_depth.lock` を排他ロック（`fcntl.flock` 等）を取得した上でチェック・更新すること (MUST)。
  - カウンタファイルが存在しない場合は `1` を書き込み、存在する場合は読み取った値をインクリメントして上書き保存すること。これらの一連の処理はロック獲得状態でアトミックに実施すること (MUST)。
  - ロックファイル内に書き込む情報には現在のプロセスID (PID) と書き込みタイムスタンプを含め、ファイルが残存していても該当PIDのプロセスが存在しない場合や一定時間（例: 5分）経過している場合は無効（深度0）とみなして自動的にリセットする生存確認ロジックを実装すること (MUST)。
  - もしカウンタの値が整数に変換できない不正な形式である場合、または負の数値である場合は、安全側の対策（Fail-safe）として閾値（5回）を超えたものとみなし、`structlog`（イベント `event='recursion_guard_triggered'` と診断メタデータ）を用いて標準エラー出力（`sys.stderr`）にエラーログを出力した上で、即座に終了コード `3` で実行を停止すること (MUST)。
  - カウンタの値が閾値（5回）を超えた場合は、`structlog`（イベント `event='recursion_guard_triggered'` と診断メタデータ）を用いて標準エラー出力にエラーログを出力し、終了コード `3` で即座に実行を停止すること (MUST)。
  - 例外発生による異常終了時や外部シグナル（SIGINT/SIGTERM等）による中断時、および正常終了時を含め、どのような終了経路をたどる場合でも、`try...finally` やシグナルハンドリングを用いてカウンタ値を安全にデクリメントするか、カウンタファイルを確実に削除（クリーンアップ）すること (MUST)。
- **パスの動的解決とバリデーション**: 
  - スクリプト内のパスは、環境変数や実行時引数、相対パスとして動的に解決できるように設計し、ソースコード内に絶対パス（特に個人ユーザー名を含むもの）を直接ハードコードしないこと (MUST)。
  - 書き込み対象となるファイル（出力レポート、JSON、一時ファイル、ロックファイル等）および入力定義ファイル（`defects.jsonl`, `target_files.json`）の絶対パスが、`Path.resolve(strict=False)` 適用後、許可されたディレクトリの共通の親パスを比較することでスコープ内であることを厳密に検証すること (MUST)。
  - もし許可ディレクトリ外への書き込み要求（ディレクトリトラバーサル等）を検知した場合は、セキュリティ例外（`ValueError`）としてトラップし、`structlog` の `ERROR` レベルで `event='SECURITY_VIOLATION_PATH_TRAVERSAL'`、`requested_path`、`allowed_scopes` を含めた構造化ログを記録した上で、終了コード `1` で即座に異常終了すること (MUST)。
  - 読み取り専用の監査対象ファイル（ソースコード等）のパスについては、プロジェクトルート外であってもシンボリックリンクを含めて正規化・解決し、読み取りを許可すること (MUST)。
- **実行環境ガード**: スクリプトの実行は、`uv run` またはアクティベートされた仮想環境（`.venv`）内で行われていること（例：環境変数 `VIRTUAL_ENV` が設定されていること、または `sys.prefix != sys.base_prefix` であること）をチェックすること (MUST)。グローバルな Python 環境で直接実行されたことを検知した場合は、`structlog` でエラーを出力した上で、終了コード `1` で即座に異常終了すること (MUST)。
- **チェックポイントの非適用**: 処理の高速性と I/O 競合回避のため、本スクリプトではチェックポイントによる状態の一時永続化は行わず、再実行時は常に最初から全件をスキャンする設計とすること (MUST)。

## 4. 開発環境・ツールチェーン要件

- **パッケージ管理**: 依存ライブラリのインストールおよび仮想環境の管理には、必ず uv を使用すること (MUST)。pip の直接使用はしてはならない (MUST NOT)。

## 5. 監査用サブエージェント共通 JSON スキーマ

次回以降の仕様書およびプログラムの監査において、L1〜L9のサブエージェントが出力する JSON 指摘配列オブジェクトは、以下のスキーマを厳格に満たすこと (MUST)。従来の指摘内容に加え、忖度なしの「ダメだし」と「改善点」を独立したフィールドとして義務付ける。

```json
    "target": "対象仕様書またはプログラムの絶対パス",

    "critic_name": "監査担当者名 (例: Resilience Critic)",
    "severity": "致命的" | "高" | "中",

  }
]
```
