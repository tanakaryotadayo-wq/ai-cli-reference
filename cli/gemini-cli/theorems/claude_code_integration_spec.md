# Claude Code 9レイヤー定理統合アーキテクチャ仕様書 (v2.0 - R1)

## 概要
本仕様書は、自律型AIコーディングエージェントである **Claude Code** に対し、本プロジェクトで確立した「エージェント工学 9レイヤー定理（L1〜L9）」を適用し、自律実行ループの信頼性、リソース効率、セキュリティ、およびエラー自己修復性能を極限まで高めるための統合アーキテクチャ設計を規定する。

## 1. 各レイヤー定理の Claude Code への適用設計

### [L1] プロンプトセマンティクス収束定理の適用 (R1修正)
*   **ユースケース**: Claude Code が大規模な Git diff やターミナル出力をコンテキストに含めて LLM を呼び出す際、プロンプトの過希釈によるアテンション散逸（指示見落とし）や、過圧縮によるセマンティクス崩壊（ハルシネーション）を防ぐ。
*   **適用モデル**:
    Claude Code の LLM 呼び出し前方に `L1SemanticsInterceptor` を配置。
    プロンプト $P$ をトークン系列の隠れ状態軌跡 $\mathcal{T}_P$ とし、アテンションカーネルの有効帯域幅 $\sigma$ を用いた意味密度分布 $\rho_P(x)$ を推定。
    有効意味情報量 $I(P)$ とトークン長 $L$ から表現密度 $\mathcal{D}(P) = I(P)/L$ を算出し、測定されたセマンティクス安定性 $\mathcal{S}(P) = \exp(-\mathbb{E}[D_{KL}])$ と比較する。
    *   **過圧縮領域 ($\mathcal{D}(P) > \mathcal{D}_c$)**: 安定性が指数関数的に減衰（$\mathcal{S}(P) \propto e^{-\alpha(\mathcal{D}-\mathcal{D}_c)}$）するため、Git diff を行単位からファイル単位に変更する等、指示の圧縮を解き具体例を補強する。
    *   **過希釈領域 ($\mathcal{D}(P) < \mathcal{D}_{opt}$)**: アテンションのSNRがべき乗則（$\mathcal{S}(P) \propto (\mathcal{D}/\mathcal{D}_{opt})^\beta$）で崩壊するため、冗長なビルドエラーログ等のノイズトークンを排除し、プロンプトを要約する。
*   **2026年技術エコシステム適合**:
    *   L8に準拠した構造化ロギング：`structlog` を用いて、`representation_density`, `measured_stability`, `status` (`Optimal` / `Over-compressed` / `Diluted`) を記録。
    *   Google ADK ライフサイクルフック統合：`pre_execution_guard` 内で `is_stable` 判定を行い、不安定（False）時は L9 (自己修復) へ `semantics_guard_violated` 例外を発行、または L7 (API縮退) へモード変更シグナルを転送する。

### [L2] 状態遷移ループフリー不動点定理の適用 (R1修正)
*   **ユースケース**: Claude Code がテスト修正やコマンド試行を行う際、同一の状態を往復する循環ループ（有向閉路）を排除し、有限ステップで目標不動点（テスト通過、エラーゼロ）に収束させる。
*   **適用モデル**:
    統合多次元状態空間 $\mathcal{V}$ の状態ベクトル $v = (s, k, \Gamma, I)$ （s: 機能状態、k: リトライ制限、Γ: APIリミット残量、I: セキュリティリスク）を定義。
    目標不動点 $v^* = (s^*, \cdot, \cdot, \cdot)$ への到達最小期待総コストに一致するポテンシャル関数 $\Phi(v)$ を導入（$\Phi(v) = 0 \iff v \in \mathcal{V}^*$）。
    統合ガード付き状態遷移システム $T_{\text{guard}}$ を構築：
    $$T_{\text{guard}}(v, e, B_{\text{L7}}) = \begin{cases} 
    v & \text{if } \Phi(v) \le \epsilon \\
    T_{\text{deg}}(v, e) & \text{if } B_{\text{L7}} = 1 \text{ or } \Phi(v) - \Phi(T_p(v, e)) < \delta_{\text{threshold}} \\
    T_p(v, e) & \text{otherwise}
    \end{cases}$$
    通常遷移 $T_p$ の進捗が $\delta_{\text{threshold}}$ 未満である場合、一様縮退収束性を持つ $T_{\text{deg}}$ に強制遷移させ、高々 $N \le \lfloor (\Phi(v_0)-\epsilon)/\min(\delta_{\text{threshold}}, \delta_{\text{deg}}) \rfloor$ ステップでの収束を保証する。ポテンシャルの単調減少性により自己ループ・閉路を完全に排除する。
*   **2026年技術エコシステム適合**:
    *   Inngest ワークフロー統合：非同期ステップ実行時に状態ベクトルのハッシュ値集合（履歴）を追跡し、循環遷移を検知した場合は即座に例外（`RuntimeError`）をスローしてサイクルを破壊。
    *   Next.js 15 & Granian 連携：`FastAPI (Granian)` からポテンシャル値 $\Phi(v)$ の推移データを返却し、Next.js 15 の Server Actions 経由で UI ダッシュボードへリアルタイムにプロット（可視化）する。

### [L3] スキル記述の直交排他定理の適用 (R1修正)
*   **ユースケース**: Claude Code が持つツール群（ファイル編集、コマンド実行、リサーチ等）の説明文（Description）が類似・重複し、モデルが誤ったツールを選択して不要な処理を実行（衝突・チャタリング）するのを防止する。
*   **適用モデル**:
    各ツール $S_i$ の説明文を完備距離空間上の単位球面 $\mathbb{S}^{d-1}$ に射影し、重心ベクトル $\mathbf{v}_i$ を定義。
    異なるツールペア $S_i, S_j$ において、許容誤判定確率を $\epsilon$ 未満に抑えるための直交排他境界を定義：
    $$d(\mathbf{v}_i, \mathbf{v}_j) > R_i + R_j + \Delta(\epsilon, \mathcal{S}(q))$$
    $$\Delta(\epsilon, \mathcal{S}(q)) = \sqrt{\sigma_{\text{model}}^2 - c \ln \mathcal{S}(q)} \sqrt{-2 \ln \epsilon}$$
    L1の実測クエリ安定性 $\mathcal{S}(q)$ が臨界値 $\mathcal{S}_{\text{crit}}$ を下回る場合は、決定論的ルーティング不可相として L7（API縮退運転・フォールバックツール化）へ遷移。
    また、正曲率球面上の sphere packing 境界から最大ツール登録数 $N_{\text{max}}$ を算出し、これを超える場合はツール定義の自動整理・統合をトリガーする。
*   **2026年技術エコシステム適合**:
    *   Google ADK ライフサイクルフック統合：`PreToolCallDecideHook` 内で `SkillOrthogonalityCritic` を駆動し、直交境界違反および限界収容数オーバーを監査。違反検知時は `L3_AUDIT_VIOLATION` 例外を発行し、L5のHITLまたはL7の縮退運転へルーティングを強制転送。
    *   Next.js 15 ダッシュボード：球面上におけるツールの埋め込みベクトル配置と、責任領域の重複状態を 3D WebGL でリアルタイム可視化。

### [L4] 自己免疫的サンドボックス隔離定理の適用 (R1修正)
*   **ユースケース**: Claude Code が任意のシェルコマンドを実行したりファイルを編集したりする際、インジェクション攻撃やホストシステムの破壊的命令を検知し、安全に実行制限・隔離を行う。
*   **適用モデル**:
    サンドボックス権限境界を $\mathcal{B}_{max} = \mathcal{B}_{min} \uplus \mathcal{P}_{destructive} \uplus \mathcal{P}_{standard}$ に分割。
    入力コマンドやコード変更ペイロードの実効リスク $I_{eff}(t) = \min(1.0, \gamma(t) \cdot I_{base}(t))$ を動的判定。
    *   **免疫適応**: 過去の侵害履歴（免疫記憶 $M(t)$）にマッチする場合（$\chi(t)=1$）は、感度 $\gamma(t)$ を増幅させ、先制的に権限を剥奪。
    *   **動的剥奪**: $I_{eff} \ge I_{crit}$ では即座に $\mathcal{B}_{min} = \{ \text{read\_telemetry}, \text{notify\_parent} \}$ まで境界を単調縮小。
    *   **ヒステリシス復元**: $K$ ステップのローリング最大リスク $\bar{I}_K(t) < I_{low}$ に低下し、監査承認 $A(t) = 1$ を得た場合のみ、段階的復元演算子 $\mathcal{R}$ を作動させて権限を再付与。パルス攻撃による権限の激しいチャタリングを完全防止。
*   **2026年技術エコシステム適合**:
    *   Granian & Google ADK 連携：ツール実行時に `execute_tool` ラッパーを介して権限をチェック。拒否時は `PermissionError` 例外を発行。
    *   Inngest ワークフロー：違反例外を `agent/security.violation` イベントとしてパブリッシュし、非同期で管理者への通知・HITLフローをトリガー。
    *   Next.js 15 監査：管理者がNext.js Server Action経由で $A(t)=1$ を送信し、FastAPI（Granian）サーバー側でサンドボックス境界を復元。

### [L5] 有向タスクグラフ最小経路定理の適用 (R1修正)
*   **ユースケース**: Claude Code が複雑なリファクタリングを自律的に継続するか、APIトークンやエラーペナルティの累積コストを懸念して人間に承認・指示（HITL）を求めるべきかの最適判断。
*   **適用モデル**:
    拡張状態空間 $V = (s, k, \Gamma, I)$ 上で期待最小コストを達成する Bellman 最最適方程式を定式化。
    自律実行成功確率 $P_{success}(e, m_e)$（L3ルーティング誤差やL4リスクに応じて減衰）が、以下の動的エスカレーション閾値 $\theta$ を下回る場合に HITL 介入（$x_e = 1$）を要請：
    $$\theta(e, k_u, m_e) = 1 - \frac{C_{hitl}(e) - C_{auto}(e, m_e)}{C_{fail}(e) + V(s_u, k_u - 1) - V(s_v, K)} \quad (k_u > 0)$$
    リトライ制限 $k_u \to 0$ に向かって閾値 $\theta$ が単調増加する特性（合理的リスク回避設計）により、失敗の繰り返しに伴って自動的かつスムーズに人間へのエスカレーションが発生する。失敗時の無限ループ（チャタリング）を排除する。
*   **2026年技術エコシステム適合**:
    *   SQLModel永続化：状態遷移コストパラメータ（$C_{hitl}, C_{auto}, P_{success}$ 等）を SQLModel で管理。
    *   Granian & FastAPI サービス：最適経路・ポリシーを後退帰納法で計算する `/optimize` エンドポイントを Granian 上でホスト。
    *   structlog ログ出力：L8と連携し、最適化イベント `optimal_path_optimized` および期待総コストを構造化ログとして記録。

### [L6] 記憶容量制限下の最適忘却定理の適用 (R1修正)
*   **ユースケース**: Claude Code のコンテキスト（会話履歴、リポジトリコード）がコンテキスト窓上限に近づき、API料金が急増したり、応答遅延やモデルの一般化能力低下が発生するのを防止する。
*   **適用モデル**:
    メモリ要素 $m_i = \langle s_i, H_i, R_i, P_i, \mathcal{S}_i \rangle$（サイズ、エントロピー、類似度、アクセス予測確率、L1安定性）を管理。
    表現密度 $\mathcal{D}_i' = H_i / (s_i \beta_i)$ に対し、過希釈と過圧縮の両面をペナルティ化するユーティリティ $\phi(\mathcal{D}_i')$ を導入。
    ラグランジュ乗数 $\lambda$ (メモリ)、$\mu$ (遅延)、$\nu$ (L2状態解像度) の下で評価関数を最大化するアクション（保持、圧縮、退避、忘却）を動的選択。
    過圧縮領域では、最適圧縮率 $\beta_i^*$ を Lambert W 関数 $W_{-1}$ を用いて解析的に決定：
    $$\beta_i^* = \frac{\gamma H_i}{-2 s_i W_{-1}\left( -\frac{1}{2} \sqrt{\frac{\lambda \gamma H_i}{K_i}} \right)}$$
    容量極限で $C < C_{min}$ の時は「状態崩壊」としてエスカレーション。遅延制限極限 $D \to 0$ では、圧縮・スワップが禁止され、「保持」か「忘却」の二者択一（LIBC）へ相転移する。
*   **2026年技術エコシステム適合**:
    *   SQLModel永続化：メモリ要素の属性（サイズ、エントロピー、アクセス履歴）を `MemoryElementEntity` としてデータベース管理。
    *   pytest & httpx テスト：`pytest + httpx` を用いて、デュアルアセント最適化アルゴリズムが一意の最適忘却状態に収束することをテスト検証。

### [L7] API制約下の縮退運転境界定理の適用 (R1修正)
*   **ユースケース**: Claude Code 実行中に Anthropic API のレートリミット（429エラー）や接続遮断が発生した場合、または時間制約（デッドライン）が迫っている場合に、ハングアップを回避して動作を継続する。
*   **適用モデル**:
    動作モード $M(t) \in \{M^{(0)}, M^{(1)}, M^{(2)}, M^{(3)}, M^{(4)}\}$（通常、ローカル推論 MLX-LM、ルールベース、HITL、FAIL）を定義。
    リスク調整後レイテンシ $\tau^{(k), safe}$ および実効待機時間 $W^{(k)}$ を用いて縮退運転境界関数を算出：
    $$\Gamma^{(k)}(t) = W^{(k)}(t) + \tau^{(k), safe} - L(t)$$
    $\Gamma^{(k)}(t) \le 0$ を満たす最小インデックスのモード $k^*(t)$ を動的に選定。
    通常モード復帰時は、前回遷移からの時間 $t - t_{trans} \ge T_{dwell}$、API補充量 $C(t) \ge N_{req} + \Delta_C$、境界余力 $\Gamma^{(0)} + \Delta_{\Gamma} \le 0$ のすべてのヒステリシス条件を要求。リアプノフ関数 $V(M(t), C(t))$ の単調非増加性により、Zeno動作（チャタリング）の完全排除とシステム安定収束を数理証明する。
*   **2026年技術エコシステム適合**:
    *   Granian & FastAPI：非同期 ASGI トークンバケットおよび境界判定ロジックを Granian サーバー上で低遅延実行。
    *   Next.js 15 SSE 配信：エージェントのモード推移および残余許容時間を Server-Sent Events (SSE) 経由で Next.js 15 管理画面へストリーミング転送し、リアルタイム表示。

### [L8] 構造化ログのエントロピー最小化定理の適用 (R1修正)
*   **ユースケース**: Claude Code の詳細な推論プロセス（思考木、ツール実行）を出力する際、ログデータの肥大化を極小化しつつ、エラーや異常発生時のデバッグに必要なトレーサビリティを完全維持する。
*   **適用モデル**:
    ログエントリ $L_k$ を、コード上の静的識別子（ASTハッシュ $ID_k$）と、動的状態の最小差分 $\Delta s_k = s_k \ominus s_{k-1}$ の直和で定義：
    $$L_k = ID_k \oplus \Delta s_k$$
    次のエントロピー最小化境界を遵守してログサイズを削減：
    $$H_{min}(L_k | L_{<k}, s_0, C) = H(s_k | s_{k-1}, C) + H(ID_k | s_k, s_{k-1}, C)$$
    静的識別子 $ID_k$ によりコード実行パスの曖昧性を $0$ にリセットし、共通コンテキスト $C$ (トレースID等) を用いて状態遷移エントロピーの上限を圧縮する。
*   **2026年技術エコシステム適合**:
    *   structlog統合：`structlog.contextvars` を用いて、共通コンテキスト（`trace_id`, `agent_id`）をスレッドローカルにバインドし、エントリごとの冗長キー出力を完全排除（エントロピー極小化）。
    *   Google ADK ライフサイクルフック統合：`L8AgentObservabilityEngine` クラスを定義。`post_tool_call` や `on_compaction` 等のフックで遷移差分 $\Delta s_k$ のみを構造化ログとして非同期出力。

### [L9] 自己修復ループの不動点収束定理の適用 (R1修正)
*   **ユースケース**: Claude Code が生成・修正したコードにコンパイルエラーやテスト不合格、静的警告（Ruffによる指摘）が検出された際、デバッグ修復ループを確実に終了（テスト合格の不動点へ収束）させる。
*   **適用モデル**:
    実行時に算出可能なランタイムポテンシャル関数 $\Phi$ を定義：
    $$\Phi(c, T(c), P_{\text{spec}}) = (M - \|T(c)\|_1) \cdot \left( w_{\text{test}} + w_{\text{sem}} \cdot d_{\text{sem}}(c, P_{\text{spec}}) \right) + w_{\text{static}} \cdot d_{\text{static}}(c)$$
    *   **不動点**: $\Phi = 0 \iff c = c^*$ (テスト全合格かつ警告0の正解コード)。
    *   **減少性**: 境界条件 $w_{\text{static}} < w_{\text{test}}$（テスト優先）および意味的非干渉境界を課すことで、毎ステップ $\Phi(c_{n+1}) \le \Phi(c_n) - \Delta$ （$\Delta > 0$）の単調減少を保証。
    *   **非循環履歴**: 生成コードのハッシュ履歴を保持し、サイクル検出時は突然変異（強制追加）を行いチャタリングから脱出。
*   **2026年技術エコシステム適合**:
    *   Google ADK 構造化出力：フィードバック時に Pydantic スキーマ（`proposed_code`, `reasoning`）による出力を強制。探索部分空間の健全な単調縮小を担保。
    *   Inngest ワークフロー：修復ステップをべき等に管理し、異常系での状態の不整合を排除。
    *   uv / Ruff / pytest 連携：`uv` 仮想環境下のサブプロセスで `Ruff`（静的解析）および `pytest`（動的テスト）を隔離実行。

## 2. 統合コヒーレンスの方程式系

各レイヤーの境界値は、以下の数理的結合方程式を通じて完全同期し、システム全体の調和的安定性を担保する。

1.  **ルーティングと成功確率 (L3 $\to$ L5)**:
    L3ルーティングの曖昧度誤差を $\epsilon(q)$ としたとき、L5における自律タスクの成功確率は以下のように制約される：
    $$P_{success} \le (1 - \epsilon(q)) P_{base}$$
2.  **セキュリティ制限と成功確率 (L4 $\to$ L5)**:
    L4の実効リスク $I_{eff} \ge I_{crit}$ のとき、サンドボックス権限が収縮するため、L5の自律成功確率は強制的に $P_{success} = 0$ となり、直ちにHITL介入閾値を下回ってエスカレーションを発生させる。
3.  **状態解像度とループフリー (L6 $\to$ L2)**:
    L6がコンテキスト窓節約のために忘却を行う際、L2のループフリー判定を保証するために必要な最小マーク解像度 $H_{crit}$ が下限境界として強制される：
    $$\sum_{i: active} \eta(a_i) H_i R_i \ge H_{crit}$$
4.  **デッドラインと遅延境界 (L7 $\to$ L6)**:
    L7のSlack Time $L(t)$ に基づき、L6のメモリ最適化で許容される最大アクセス遅延境界 $D_t$ が動的に切り詰められる。デッドライン直前（$D_t \to 0$）では、スワップや再計算を伴う圧縮が禁止され、保持か忘却かのバイナリ選択（LIBC）へ相転移する。
