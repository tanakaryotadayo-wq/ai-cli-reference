# verify_theorems.py 仕様書 (v10)

## 概要
AIエージェント工学の3つの定理候補（散逸停止定理、曖昧性収縮定理、最小作用ワークフロー定理）について、すでに監査・検出された欠陥データ（`defects.jsonl`）および対象ファイルリストを用いて、定理の成立条件（再現性、影響度、反例比率）を統計的に集計・検証する Python スクリプトの実装仕様書。

各定理の最終的な「合格（成立）」判定は、再現性、影響度、反例比率のすべての検証基準を同時に満たした場合（論理積：AND）のみ合格とし、いずれか一つでも満たさない場合は不合格とすること (MUST)。

また、検証結果のレポート出力および監査エージェントの指摘スキーマにおいて、忖度なしの「ダメだし (criticism)」と「改善点 (improvement)」を出力することを義務付ける (MUST)。

さらに、Wick-Loop インスタントン遷移モデルにおける有界脱出の安全性および乱数ストリームの因果分離に関する追加定理（定理 V2.2）および実装ルールを本仕様書に統合する。

## 1. 動作要件 (Functional Requirements)

### データ入力
- **データ入力**: コマンドライン引数 `--defects` で指定されたパス（デフォルト: `$APP_DATA_DIR/scratch/defects.jsonl`）から欠陥データファイル（`defects.jsonl`）を1行ずつ読み込み、パースすること (MUST)。また、検証対象となるファイルリストは、コマンドライン引数 `--targets` で指定された JSON ファイル（デフォルト: `target_files.json`）から読み込むこと (MUST)。
- **入力パラメータのスキーマバリデーション**: 引数 `--workflow-context` に対し、英数字、ハイフン、アンダースコア、ピリオドのみで構成される文字列であるかを検証し、これに違反する場合はセキュリティ例外として終了コード `1` で異常終了すること (MUST)。また、`--targets` で指定された JSON ファイルのパース直後に、その内容が文字列の配列（ファイルパスのリスト）であることを厳密に検証し、形式が異なる場合はエラーログを出力し終了コード `1` で終了すること (MUST)。
- **入力サイズ制限（DoS対策）**: `defects.jsonl` を1行ずつ読み込む際、1行あたりの最大許容サイズ（10MB）を設け、これを超える巨大な行を検出した場合は、セキュリティ保護のため処理を中断し、終了コード `1` で異常終了すること (MUST)。
- **メモリフットプリント制限と一時ファイル退避**: パース後の詳細なオブジェクトをすべてメモリ上に保持してはならず (MUST NOT)、集計用の最小限のメトリクス（正規化されたファイルパスごとの `pattern_class` カウント、深刻度比率、レイヤーごとの深刻欠陥の有無、および反例判定フラグ・反例ファイルリスト）のみを状態変数としてメモリに保持すること (MUST)。
  - ただし、メモリに保持する反例ファイルパスの最大件数（履歴バッファ上限）は 1,000件 とし (MUST)、これを超える件数の反例が検出された場合は、メモリの破綻を防ぐため、追加の反例パス情報を一時ファイル（所有者のみ読み書き可能な `0600` 権限の一時バッファファイル、例: `counterexamples.tmp`）に順次ストリーム出力して退避（コンパクション）させ、件数のみをメモリ上でカウントし続けること (MUST)。
  - 上限を超えて一時ファイルに退避した場合は、検証結果の JSON メタデータおよび Markdown レポートの生成時に一時ファイルからデータをバッチで読み出してマージし、リストが切り捨てられたことを示すメタデータ（`counterexamples_truncated: true`）と警告メッセージを必ず含めるとともに、一時ファイルを確実にクリーンアップ（削除）すること (MUST)。

### 定理の統計的検証
- **再現性 (Reproducibility)**: 各定理に対応する以下の `pattern_class` が、監査対象の全ファイル（読み込まれたパスリスト of 総件数。デフォルト88件）中 **30%以上** で検出されていることを検証すること (MUST)。ただし、当該定理に対応する欠陥が全ファイルで1件も検出されなかった場合（分母が0の場合）は、再現性基準を自動的に満たしたもの（合格）として処理すること (MUST)。
  - 散逸停止定理: `missing_stop_rule` であることを検証すること (MUST)。
  - 曖昧性収縮定理: `missing_golden_rule` であることを検証すること (MUST)。
  - 最小作用ワークフロー定理: `missing_execution_workflow` または `missing_workflow` のいずれか一方（あるいは両方）が1件以上検出されているファイルを対象とし、該当するユニークなファイル数の論理和（OR）として分子をカウントし、検出率を計算すること (MUST)。
- **影響度 (Impact)**: 各定理に対応する検出された欠陥のうち、深刻度（`severity` フィールドの値が `"高"` または `"致命的"`、あるいは `"high"` または `"critical"`）である割合（深刻度比率）が **50%以上** であるかを検証すること (MUST)。
  - 最小作用ワークフロー定理においては、`missing_execution_workflow` と `missing_workflow` の欠陥を同一の定理カテゴリとして合算（マージ）した欠陥集合を対象として集計を行うこと (MUST)。
  - ただし、対応する欠陥が1件も検出されていない場合（分母が0）は、後述の「欠陥数ゼロの例外処理」に従って判定し (MUST)、それ以外のパース例外やデータ破損等に起因して予期せぬゼロ除算が発生した場合は、安全側の処理として深刻度比率を 0% とし、当該定理は不合格と判定すること (MUST)。
- **反例検証 (Counterexample Analysis)**: 各定理に対応する `pattern_class` が検出されている（＝ルールが未定義である）ファイル群を分母とし、その中で、L2 または L4 レイヤー（`layer` フィールドの値が `"L2"` または `"L4"`）かつ深刻度（`severity` フィールドの値が `"高"` または `"致命的"`、あるいは `"high"` または `"critical"`）である欠陥が1件も検出されていないファイル（＝反例）の割合（分子 / 分母）が **10%以下** であるかを判定すること (MUST)。
- **欠陥数ゼロの例外処理（分母ゼロ）**:
  - データパースに成功し、無効データのない正常な処理下において、該当する欠陥が1件も検出されず「分母が0」となった場合は、プロジェクトが当該欠陥に関して完全に健全であるとみなし、再現性・影響度・反例比率のすべてについて基準を満たしたもの（合格）として判定し (MUST)、自動修復のための HITL 警告（`hitl_required` = `false`）とすること (MUST)。
  - 一方で、入力された defects.jsonl が0バイトであるか、または defects.jsonl のパースで有効なレコードが1件も取得できなかったなどの例外的な処理エラーにより「すべての定理の分母が0」となった場合は、データ破損による異常とみなし、Markdownレポート内にオペレータの確認が必要な旨の警告項目「HITL確認要求」を明記し (MUST)、`structlog` を用いて `hitl_required=True` メタデータを含む警告ログ（`WARNING` レベル、イベント名 `DATA_CORRUPTION_DETECTED`）を出力し、オペレータに確認を促すこと (MUST)。（※この警告発生に伴い、後述の `HITL_PENDING_APPROVAL` イベントの構造化ログも出力すること）

### ダメだし (criticism) と 改善点 (improvement) の義務化
- **検証プログラムの出力結果における要件**:
  - `theorem_verification_results.json` (JSON) および `theorem_verification_results.md` (Markdownレポート) において、各定理の判定結果を出力すること (MUST)。
  - JSONファイル内では、キー名として `criticism` および `improvement` を使用すること (MUST)。外部のLLM呼び出し等は行わず、スクリプト内の数値条件分岐に基づいたルールベースの定型テキスト生成ロジックとして完結させること (MUST)。
  - Markdownレポート内では、見出し名として「ダメだし (Criticism)」および「改善点 (Improvement)」を使用すること (MUST)。
    - **`criticism` (ダメだし)**: 対象定理の検証基準が未達成となった理由や、データに含まれる具体的な欠陥傾向について、集計されたメトリクスの値に応じて事前に定義されたテンプレート文から動的に生成する客観的な指摘文章 (MUST)。
    - **`improvement` (改善点)**: 検証基準をクリアするために、AIエージェントの行動仕様、設計ルール、またはデータ収集プロセスをどのように変更すべきかを、各メトリクスの未達成状況に応じた具体的なアクションアイテムとして提示する文章 (MUST)。
  - 各定理のステータスに応じた定型テキストの定義 (MUST):
    - **すべての検証基準を達成（合格）した場合**:
      - criticism: `'すべての検証基準を満たしていますが、将来的な仕様変更時の回帰を防止するため、継続的な監視を推奨します。'`
      - improvement: `'現在の行動仕様および設計ルールを維持し、定期的な監査を実行してください。'`
    - **いずれかの基準が未達成（不合格）の場合**:
      - 再現性、影響度、反例比率の複数が同時に未達成となった場合は、未達成となったすべての基準に対応する `criticism` および `improvement` の定型テキストを改行（`\n`）で結合して出力すること (MUST)。単一の未達成の場合は以下の該当するテキストのみを出力すること (MUST)。
      - 再現性未達: criticism=`'定理の再現性基準（30%以上）に達していません。'`, improvement=`'対象のルール（例: stop_rule, golden_rule）が定義されていないファイルの割合を減らすため、エージェントの基本指示に該当ルールを明記してください。'`
      - 影響度未達: criticism=`'検出された欠陥における深刻度（高・致命的）の割合が50%未満です。'`, improvement=`'深刻な欠陥の発生を抑えるため、該当する設計レイヤー of バリデーションを強化してください。'`
      - 反例比率超過: criticism=`'深刻な欠陥が検出されなかったファイル（反例）の割合が10%を超えています。'`, improvement=`'例外ケースのハンドリングルールを見直し、反例となる健全なファイルでもルール遵守が徹底されるよう設計を修正してください。'`
    - **データ破損や縮退運転により検証不可（UNVERIFIABLE）の場合**:
      - criticism: `'データ破損、パースエラーの多発、または入力ファイルサイズが0であるため、当該定理の検証を実行できませんでした。'`
      - improvement: `'入力データである defects.jsonl の整合性を確認し、パースエラーの原因を修正した上で再実行してください。'`
  - 定理が合格（成立）している場合でも、さらなる堅牢性向上の観点から「ダメだし (criticism)」と「改善点 (improvement)」を記載すること (MUST)。

### データ出力
- **レポート出力の整合性保護**: 検証結果のメトリクス、合格/不合格判定、および「ダメだし」「改善点」をまとめたレポートを、コマンドライン引数 `--output` で指定された出力パス（デフォルト: `$APP_DATA_DIR/scratch/theorem_verification_results.md`）に書き出すこと (MUST)。
- **ファイル出力の状態遷移シーケンス**: ファイル出力（MarkdownおよびJSON）を行う際は、ターゲットパスおよびバックアップパスがシンボリックリンクでないこと（`is_symlink()` が偽であること）(MUST)、および解決された絶対パスが許可されたディレクトリにあることを検証した上で、同一ファイルシステム上でのアトミック性を保証するためターゲットファイルと同一のディレクトリ内に一時ファイル（ターゲットパスに拡張子 `.tmp` を付与したもの）を作成し、以下の順序でアトミックに実行すること (MUST)。
  1) 一時ファイルに検証結果を書き出すこと (MUST)。このとき、所有者のみ読み書き可能なアクセス権限（`0600`）を明示的に指定して作成すること (MUST)。
  2) 書き出しが成功した場合、既存のレポートファイルが存在すれば、それがシンボリックリンクでないことを確認した上で、バックアップパス（ターゲットパスに拡張子 `.bak` を付与したもの）へコピーして退避させること (MUST)。このバックアップファイルに対しても、所有者のみ読み書き可能なアクセス権限（`0600`）を明示的に指定すること (MUST)。もしこのバックアップコピー処理が I/O エラー等で失敗した場合は、安全のためターゲットファイルの置換は行わず、作成中の一時ファイルを削除した上で、エラーログを記録して終了コード `1` で異常終了すること (MUST)。
  3) 一時ファイルをターゲットパスへアトミックにリネーム（置換）すること (MUST)。同一デバイス境界外エラー（EXDEV）などの理由でリネームが失敗した場合は、安全にコピーおよび削除でフォールバックすること (MUST)。このフォールバック処理中に書き込みエラーや容量不足等の例外が発生した場合は、書きかけの不完全なターゲットファイルを削除してクリーンアップした上で、バックアップファイル（`.bak`）が存在する場合はそれをターゲットパスに復元し、作成中の一時ファイルを確実に削除（クリーンアップ）し、エラーログを記録して終了コード `1` で異常終了すること (MUST)。
  4) 処理中にエラーが発生し中断した場合は、作成中の一時ファイルを確実に削除（クリーンアップ）すること (MUST)。
- **ファイルパーミッション制限**: レポートファイルおよび一時ファイルを作成する際は、所有者のみ読み書き可能なアクセス権限（`0600`）を明示的に指定して作成すること (MUST)。出力先ディレクトリが存在しない場合は、所有者のみ読み書き実行可能な権限（`0700`）で自動作成すること (MUST)。
- **インジェクション防止**: レポート生成時、`defects.jsonl` から取得したテキストデータ（`criticism` や `improvement` 等の任意の文字列）を埋め込む際は、インジェクション対策として、共通の安全なマークダウン生成ユーティリティ（あるいは Jinja2 などのテンプレートエンジン）の自動エスケープ機能を使用するか、HTML/マークダウンの特殊文字が安全に無効化される共通エスケープ処理を適用すること (MUST)。
- **JSONメタデータ出力**: 後続のAIエージェントや検証パイプラインが自己反映（Self-Reflection）および自動修復のアクションを実行できるよう、検証結果の詳細（各定理の判定（合格時は `"PASS"`、不合格時は `"FAIL"`、検証不可時は `"UNVERIFIABLE"` を設定する `result` フィールド）、メトリクス数値、反例ファイルリスト、`criticism`、`improvement`、`counterexamples_truncated` (Boolean型)、`hitl_required` (Boolean型)、およびコマンドライン引数 `--workflow-context` で指定された任意の文字列）を含む構造化された JSON ファイルを、コマンドライン引数 `--output-json` で指定されたパス（デフォルト: `$APP_DATA_DIR/scratch/theorem_verification_results.json`）に書き出すこと (MUST)。
  - `hitl_required` フラグが `true` の場合、後続の自動修復パイプラインは自動実行を中断し、オペレータの確認を待つこと (MUST)。
  - `hitl_required` が `true` の場合は、`structlog` にて `event='HITL_PENDING_APPROVAL'` という一意のイベント名を持つ構造化ログを標準エラー出力（`sys.stderr`）に出力し、そのログ内に `workflow_context` も含め、後続のパイプラインやCI/CD環境がそのログイベントをキャッチして Slack や GitHub PR 等に承認依頼を自動通知できるようにシグナリング仕様を統合すること (MUST)。
  - JSON出力に含まれる動的テキストフィールド（`criticism` や `improvement` 等）については、後続のAIエージェントにおけるプロンプトインジェクション攻撃を防止するため、安易なブラックリスト方式のフィルタに依存せず、データを構造的（マークダウンやXMLタグ等で囲むなど）にカプセル化し、後続のエージェントがシステムプロンプトのコンテキストと検証データを安全に分離・処理できるように構造的カプセル化ロジックを実装すること (MUST)。データと命令の明確な分離を保証すること。
  - `counterexamples_truncated` が `true` の場合、後続の自己反映・自動修復パイプラインは全件の修復が1サイクルで完了しないと判断し、一部のファイルのみが修復されることによる整合性破壊を防ぐために、強制的に `hitl_required` を `true` に設定して、処理をオペレータの確認待ち（HITL）にすること (MUST)。
- **非対話的実行原則**: スクリプトの実行は常に非対話的（Non-blocking）に行うものとし、警告ログを出力した後にオペレータの入力を待って一時停止してはならない (MUST NOT)。警告は出力およびファイル記録のみにとどめること (MUST)。

## 2. Task Execution Workflow (実行手順)

### オペレーター/エージェント作業ワークフロー

1. **事前準備 (MUST)**
   - 監査対象のパスリストが記述された JSON ファイル `target_files.json` を作成し、プロジェクトのルートディレクトリまたは指定された作業ディレクトリに配置すること (MUST)。
   - 欠陥データ `$APP_DATA_DIR/scratch/defects.jsonl` が存在し、読み取り可能であることを確認すること (MUST)。
   - 実行環境に `uv` がインストールされており、仮想環境が有効化されていることを確認すること (MUST)。

2. **実行コマンドと引数指定 (MUST)**
   - スクリプトは以下のコマンドで非対話的に実行すること (MUST)。
     ```bash
     uv run verify_theorems.py \
       --targets target_files.json \
       --defects $APP_DATA_DIR/scratch/defects.jsonl \
       --output $APP_DATA_DIR/scratch/theorem_verification_results.md \
       --output-json $APP_DATA_DIR/scratch/theorem_verification_results.json \
       --workflow-context "任意の一意な実行識別コンテキスト文字列"
     ```
   - デバッグ時は `--debug` オプションを付与し、処理の各ステップやパース詳細をログ出力すること (MUST)。

3. **実行中進捗の監視 (MUST)**
   - 標準出力および標準エラー出力（`sys.stderr`）に出力される構造化ログを監視し、`event='WORKFLOW_PHASE_CHANGED'` のログから現在の進捗フェーズ（`\"setup\"`, `\"mapping\"`, `\"statistics\"`, `\"reporting\"`）の遷移および `status='completed'` への遷移を自動検出すること (MUST)。
   - 各フェーズの実行が 30 秒以上遷移しない場合、またはスクリプト全体の実行が開始から 120 秒を超えた場合は、ハングアップと判定して処理をタイムアウト（異常系停止）させ、一時ファイルを削除して終了すること (MUST)。
   - もしリトライログ（`event='IO_RETRY_ATTEMPTED'`）が出力された場合は、エラー内容を確認しつつ、スクリプトの指数バックオフによる自動リトライ処理（最大5回）の推移を静観すること (MUST)。

4. **正常終了時の検証とクリーンアップ (MUST)**
   - 終了コード `0` で完了したことを確認すること (MUST)。
   - 出力された JSON メタデータ `theorem_verification_results.json` および Markdown レポート `theorem_verification_results.md` が正常に生成され、シンボリックリンクでないこと、パーミッションが `0600` であることを確認すること (MUST)。
   - 一時ファイル（`.tmp`）が残存せず、正しく削除されていることを確認すること (MUST)。
   - 排他ロック・カウンタファイル `$APP_DATA_DIR/scratch/execution_depth.lock` が正常にデクリメントまたは削除されていることを確認すること (MUST)。

     2) `defects.jsonl` の該当箇所を修正して JSONL のフォーマットを満たすようにすること (MUST)。
   - **終了コード 3（再帰制限超過）**:

## 3. 動作安全性 (Safety Rules)

- **構造化ログの適用**: スクリプト内のすべてのログ出力（進捗、デバッグ、エラー）は、`structlog` を使用して構造化 JSON 形式で出力すること (MUST)。標準の `logging` や `print` は使用してはならない (MUST NOT)。また、構造化ログ（structlog）の出力チャネルは、標準エラー出力（`sys.stderr`）に統一すること (MUST)。
- **情報のフィルタリング（マスク処理）**: ログ出力処理において、環境変数や引数から取得したパス、シークレットなどの機密情報がそのままログに出力されないよう、起動時に動的にカレントユーザー名（例: `getpass.getuser()` で取得される文字列）や `APP_DATA_DIR` のパスを検出し、`structlog` のカスタムプロセッサを通じて、ユーザー名部分を `[USER]` に、`APP_DATA_DIR` のパス部分を `[APP_DATA_DIR]` にそれぞれ置換するマスク処理や、機密情報（APIトークン、パスワードなど）を `[REDACTED]` にマスクする処理を実装すること (MUST)。
- **デバッグログの制御**: コマンドライン引数で `--debug` が指定された場合は、`structlog` の最小ログレベルを `DEBUG` に動的に変更すること (MUST)。デバッグモードが有効な場合、各データ行のパース開始・終了、正規化された絶対パスの検証結果、および各定理の集計過程におけるメトリクス更新状況などの診断トレースを `DEBUG` レベルの構造化ログとして出力すること (MUST)。デバッグ引数がない場合は、ログレベルを `INFO` 以上に制限すること (MUST)。
- **進捗ログの標準化**: 進捗ログはタイムスタンプの出力形式として ISO 8601（UTC または ローカルタイムゾーン付きミリ秒精度）を強制し、`structlog.processors.TimeStamper(fmt="iso")` の使用を指定すること (MUST)。また、進捗フェーズのログでは、`event` フィールドに `"WORKFLOW_PHASE_CHANGED"` を設定し、`phase` キーの値（以下のフェーズ定義）およびフェーズの遷移状態を示す `status` キーの値（`\"started\"` | `\"completed\"` | `\"failed\"`）を必ず出力すること (MUST)。
  - 特に、例外発生やバリデーション違反等でスクリプトが異常終了（終了コード1, 2, 3）する場合は、終了直前に現在のアクティブなフェーズに対して `status=\"failed\"` とエラー内容（`error_class`, `error_message`）を付与した `WORKFLOW_PHASE_CHANGED` ログを出力すること (MUST)。
  - `\"setup\"`: スクリプト起動、環境変数・引数パース、再帰呼び出しカウンタの検証および初期化処理。
  - `\"mapping\"`: ターゲットファイルリストおよび欠陥データの読み込み、絶対パスへの正規化、およびファイルごとの欠陥マッピング処理。
    - データの読み込みおよびマッピング処理中は、進捗フィードバックとして、処理されたレコード数またはターゲットファイル数が全体の 10% 増加するごとに、`event='PROCESSING_PROGRESS'`、`processed_count`、`total_count`、および `progress_percentage` を含む構造化ログを出力すること (MUST)。
  - `\"statistics\"`: 各定理に対応する統計メトリクスの集計および検証判定の計算処理。
  - `\"reporting\"`: MarkdownおよびJSONレポートの一時ファイル書き出し、バックアップ作成、アトミック置換、およびカウンタファイルのクリーンアップ処理。
- **Golden Rule (デフォルト動作と縮退運転)**: 
  - **エージェント実行UXのデフォルト動作**: エージェントが実行引数の解釈や対話プロンプト（HITL）の応答待ちにおいて、判断に迷うか応答が得られない場合（タイムアウト：180秒）は、独自の推測で処理を続行せず、安全側に倒して処理を即座にキャンセル（一時ファイルをクリーンアップ）し、終了コード `1` で異常終了すること (MUST)。
  - データパース時において、JSONデコードエラー、あるいは必須キー（`target`, `pattern_class`, `severity`, `layer`）が欠落している行を「無効データ」と判定し、その行を警告ログ（レベル: `WARNING`）として記録した上でスキップし、残りのデータのパース処理を継続すること (MUST)。警告ログにはエラーが発生した「ファイルパス」「行番号」「生データの抜粋（マスク処理後）」「パース失敗理由（例外メッセージ）」を必ず含めること (MUST)。
    - 生データの抜粋（マスク処理後）においては、シークレット（APIキー、トークン、パスワード等）に合致する文字列を `[REDACTED]` にマスクするほか、ログの肥大化と機密情報の偶発的漏洩を防ぐため、抜粋する文字数は該当行の先頭から最大100文字に制限すること (MUST)。
  - パース処理の開始前に対象ファイルの総行数（総レコード件数）をプリスキャンして取得し、パース中に検出された無効データ件数の累積がその 5% を超えた時点で、初めてデータの健全性が損なわれていると判断し、すべての定理を対象として『データ破損により検証不可 (UNVERIFIABLE)』と判定すること (MUST)。
  - 累積の無効データ件数が 5% 以下である場合は、JSONデコードエラーや `pattern_class` が特定できない行が含まれていても、それらを単に無視して残りの有効なデータ行のみを用いて統計処理を継続（縮退運転）すること (MUST)。この際、検証不可にならなかった定理については通常の判定（合格/不合格）を出力し、`hitl_required` は `false` のままとすること (MUST)。
  - 無効データが 5% を超えてすべての定理が検証不可となった場合は、それまでに検出された全パースエラーのサマリーを `structlog` を用いて一括出力した上で、強制的に `hitl_required` を `true` に設定し、`event='HITL_PENDING_APPROVAL'` の構造化ログを標準エラー出力に出力して終了コード `2` で異常終了すること (MUST)。
- **Stop Rule (異常停止条件と指数バックオフ)**: 
  - **エージェント実行・リカバリの異常停止基準**: エージェントによるスクリプトの実行、または異常終了時のリカバリ操作（コマンド実行やファイル編集等）において、エラーまたは競合が連続して 5 回以上発生した場合は、即座にすべての自動処理・変更操作を停止し、発生したエラーログを要約してオペレータに報告し、その後の指示を仰ぐこと (MUST)。
  - ファイルの読み込みまたは書き込みにおいて、一時的な I/O エラー（`BlockingIOError`, `TimeoutError`）が発生した場合は、ジッターを伴う指数バックオフ（初期待ち時間 0.5 秒、乗数 2、最大待ち時間 5 秒、最大リトライ回数 5 回）を適用したリトライループを実装すること (MUST)。
  - リトライ試行の都度、`structlog` を用いて一意のイベント名 `event='IO_RETRY_ATTEMPTED'` とともに、現在の試行回数 `attempt`、最大試行回数 `max_attempts`、次の試行までの待ち秒数 `backoff_duration`、および発生した一時例外メッセージを含む構造化ログを標準エラー出力（`sys.stderr`）に即座に出力すること (MUST)。
  - 回復不能な I/O エラー（`PermissionError`, `FileNotFoundError`, `IsADirectoryError` など）、またはリトライ回数を超過した場合は、即時に実行を停止し、`structlog` で例外情報（`exc_info=True` を指定し、適切にフォーマットされたスタックトレース）と、マスク処理済みの詳細な診断コンテキスト情報（エラーが発生したファイルパス、実行中のアトミックシーケンスフェーズ、処理中行番号、リトライ試行回数等）を `ERROR` または `CRITICAL` レベルで標準エラー出力（`sys.stderr`）に出力した上で、終了コード `1` で異常終了すること (MUST)。スタックトレース出力時も情報のフィルタリング要件を適用し、シークレットや認証情報を確実にマスクすること (MUST)。
- **再帰呼び出しガード**: 
  - エージェント実行環境下におけるスクリプトの無限再帰呼び出しを防止するため、環境変数 `THEOREM_VERIFICATION_DEPTH` を使用して呼び出し深さを管理すること (MUST)。スクリプト起動時に同環境変数の値を読み込み、値が存在しない場合は `0` とみなし、存在する場合は整数に変換すること (MUST)。

  - 検証処理の過程でスクリプトが自身をサブプロセスとして再帰的に呼び出す必要がある場合は、必ず現在の環境変数の値をインクリメントした値を子プロセスの環境変数 `THEOREM_VERIFICATION_DEPTH` に設定して起動すること (MUST)。

  - ロック獲得後、ファイルから既存のカウンタ値、書き込みPID、およびタイムスタンプの読み取りを試みること (MUST)。ファイルが空または新規作成直後の場合はカウンタ値を 0 としてみなすこと (MUST)。該当PIDのプロセスが存在しない場合、または書き込みから 5 分以上経過している場合は、ロックファイルが残存していても状態が無効（深度0）とみなして自動的にカウンタ値を0にリセットすること (MUST)。その後、生存が確認されたカウンタ値をインクリメントしてファイルを上書き保存し、ファイルをフラッシュ（`flush`および`os.fsync`）すること。取得した排他ロックは、本処理がすべて終了し、`finally` ブロックでカウンタ値をデクリメントして書き込みを完了するまで保持し続けること (MUST)。

  - スクリプト内のパスは、環境変数や実行時引数、相対パスとして動的に解決できるように設計し、ソースコード内に絶対パス（特に個人ユーザー名を含むもの）を直接ハードコードしないこと (MUST)。

## 5. Wick-Loop 因果分離・有界脱出定理 V2.2

本節では、V2.1 のインスタントン遷移モデルを、単発の成功軌跡ではなく、有限回の試行における脱出確率・失敗確率・期待コストとして再定式化する。

### 定理 V2.2：Wick-Loop 因果分離・有界脱出定理

#### 定義
ループ検知後の第 n 回目のトンネル試行における有効ひらめき度を

$$\hbar_n = \min\left(\hbar_0(1+\alpha n), \hbar_{\max}\right)$$

とする。

そのとき、作用 $S_E$ を持つ障壁に対する第 n 回目のトンネル成功確率を

$$p_n = A \exp\left(-\frac{S_E}{\hbar_n}\right)$$

と定義する。

ここで、
* $A$ は前因子。
* $0 < A \leq 1$。
* $n=0,1,2,\ldots,N_{\max}-1$。
* $N_{\max}$ は最大試行回数（デフォルト $N_{\max} = 9$ または $10$）。

#### 主張 1：有界脱出確率
最大 $N$ 回まで試行する場合、少なくとも一度トンネルに成功する確率は

$$P_{\leq N} = 1-\prod_{n=0}^{N-1}(1-p_n)$$

で与えられる。

失敗確率は

$$P_{\text{fail}}(N) = \prod_{n=0}^{N-1}(1-p_n)$$

である。

したがって、適応的 $\hbar$ 制御下では、$\hbar_n$ が単調増加する限り、$p_n$ も単調増加し、失敗確率は各試行ごとに急速に減少する。

#### 主張 2：固定 $\hbar$ に対する優越性
$\alpha > 0$ のとき、

$$\hbar_n \geq \hbar_0$$

なので、

$$p_n \geq p_0$$

が成り立つ。

したがって、

$$P_{\leq N}^{\text{adaptive}} \geq 1-(1-p_0)^N$$

である。

つまり、適応的ひらめき補正は、同じ初期 $\hbar_0$ を使う固定モデルより、有限試行回数内の脱出確率を必ず改善する。

#### 主張 3：コスト支配条件
Wick-Loop ワープを実行する総期待コストを

$$C_{\text{WL}} = C_{\text{detect}} + c_{\text{eval}}\sum_{n=0}^{N-1}Q_n + C_{\text{safe}}Q_N$$

とする。

ここで、

$$Q_n=\prod_{j=0}^{n-1}(1-p_j)$$

は第 n 回目の試行まで到達する確率。

古典的リトライの期待コストを $C_{\text{classic}}$ としたとき、

$$C_{\text{WL}} < C_{\text{classic}}$$

を満たす場合に限り、Wick-Loop ワープは実行すべきである。

これにより、従来の「APIコストゼロ」という過大表現は不要になる。正確には、Wick-Loop は API コストをゼロにするのではなく、無意味な線形リトライ列を有限回の検証コストに置換し、期待コストを下げる制御則である。
